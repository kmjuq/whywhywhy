# 智能文本分析工具 - 思路轨迹记录

## 概述
本文档记录项目开发过程中的问题、决策和思路演变，便于回溯和参考。

---

## 决策记录

### 2026-06-09

#### 1. 初始需求定义
**问题**：开发一个桌面小软件，可以托盘到操作系统，滑词/滑段落生成问题标签，点击查询大模型，支持翻译，支持多平台（mac、windows、linux）

**分析**：
- 核心功能：系统托盘、文本捕获、问题标签生成、大模型查询、翻译
- 目标平台：macOS、Windows、Linux
- 需要选择跨平台框架

**决策**：选择 Tauri 作为跨平台框架（轻量级、性能好），Vue3 作为前端框架

---

#### 2. 问题标签生成方式
**问题**：问题标签应该如何生成？

**分析**：
- 方案A：程序规则生成 - 固定模板，无法适应不同领域
- 方案B：大模型生成 - 根据文本内容动态生成，自适应领域

**决策**：采用方案B，通过大模型生成问题标签，因为不同领域的问题重点不同，需要领域自适应

---

#### 3. 前端框架选择
**问题**：选择 React 还是 Vue3？

**分析**：
- React：生态成熟，资源丰富
- Vue3：组合式API，更简洁的语法，学习曲线平缓

**决策**：使用 Vue3 + TypeScript，配合 Pinia 状态管理

---

#### 4. 项目定位修正
**问题**：项目背景和目标用户描述不准确

**分析**：
- 原描述："帮助用户快速分析文本内容"
- 实际需求："快速扫盲"，了解知识点的基本信息（是什么、干什么用、怎么用）

**决策**：重新定义项目定位为"快速扫盲工具"，聚焦基础问题和简洁回答

---

#### 5. 界面设计方案
**问题**：如何设计主界面？

**分析**：
- 方案A：标签云展示问题标签
- 方案B：Mini弹窗，分区域展示

**决策**：采用方案B，Mini弹窗设计：
- 上段：问题列表区（大模型生成的问题）
- 下段：问题编辑与发送区（可编辑补充）
- 答案区：内嵌于弹窗，展开显示

---

#### 6. 托盘菜单设计
**问题**：托盘菜单应该包含哪些选项？

**参考**：easydict 设计

**决策**：菜单结构：
- 分析文本（打开主弹窗）
- 历史记录
- 设置...

#### 7. 功能裁剪（迭代规划）
**问题**：哪些功能可以延后迭代？

**分析**：
- 翻译功能：需求不明确，可延后
- 文本输入区：通过剪贴板捕获已足够，可移除
- Touch Bar：新MacBook已无此功能，可移除

**决策**：
- 移除翻译功能（后续迭代）
- 移除文本输入区（仅保留剪贴板捕获）
- 移除 macOS Touch Bar 支持

#### 8. 文本捕获功能实现方案
**问题**：如何实现剪贴板监听？

**分析**：
- 方案A：仅使用事件监听（clipboard://text-changed）
- 方案B：事件监听 + 轮询机制（双重保障）
- 方案C：仅使用轮询机制

**决策**：采用方案C，使用纯轮询机制（500ms间隔）。

**技术实现**：
- 使用 `@tauri-apps/plugin-clipboard-manager` 读取剪贴板
- 每 500ms 检查一次剪贴板内容
- 添加去重逻辑，避免重复捕获相同文本
- 移除了不可靠的事件监听

**遇到的问题及解决**：
- 初始实现使用事件监听 `clipboard://text-changed`，但该事件在某些平台上不可靠
- 改为轮询机制后，虽然简单但更可靠
- 使用 fake timers 进行单元测试，确保测试的稳定性
- **Tauri 2.x 权限问题**：前端调用 `clipboard-manager.read_text` 时报错权限不足
  - 解决：在 `src-tauri/capabilities/default.json` 中添加明确权限
  - 添加的权限：`clipboard-manager:allow-read-text`, `clipboard-manager:allow-write-text`, `global-shortcut:allow-*` 等
  - **2026-06-09 验证**：功能正常工作，界面可以显示捕获的文本 ✅

#### 9. 问题标签生成功能实现
**问题**：如何实现基于大模型的问题标签生成？

**分析**：
- 需要调用大模型 API 生成问题
- 问题类型应该聚焦"快速扫盲"
- 需要支持配置 API Key 和模型选择

**技术实现**：
- 创建 `useLLM.ts` composable
- 使用 OpenAI ChatGPT API
- 支持配置 API Key 和模型选择
- 本地存储配置（localStorage）
- 生成 5-8 个基础问题标签

**问题生成提示词设计**：
```
请分析以下文本内容，生成5-8个适合"快速扫盲"的基础问题标签。
问题类型应包括：
- 这是什么？（定义、概念、本质）
- 干什么用的？（用途、作用、功能）
- 怎么用？（使用方法、基本操作）
- 有什么特点？（特性、优势、亮点）
```

**UI 界面设计**：
- 捕获文本显示区
- 问题标签列表（可点击）
- 问题输入框（可编辑）
- 答案展示区
- 设置弹窗（API Key、模型选择）

**单元测试**：
- 创建 9 个测试用例
- 覆盖 API Key 管理、问题生成、答案查询等
- 所有测试通过 ✅

#### 10. DeepSeek 多服务商支持
**问题**：用户需要支持 DeepSeek API

**分析**：
- DeepSeek 是国产优秀的大模型
- 成本较低，性能优异
- 需要支持多服务商切换

**技术实现**：
- 添加 LLMProvider 接口和预定义服务商列表
- 支持 OpenAI 和 DeepSeek 两个服务商
- 动态获取服务商对应的模型列表
- 使用 localStorage 存储服务商和 API Base 配置
- 支持自定义 API Base URL（为未来扩展做准备）

**支持的模型**：

| 服务商 | 模型 | API Base |
|-------|------|---------|
| DeepSeek | deepseek-chat, deepseek-coder | https://api.deepseek.com/v1 |
| OpenAI | gpt-4o-mini, gpt-4o, gpt-4-turbo, gpt-3.5-turbo | https://api.openai.com/v1 |

**UI 更新**：
- 设置面板添加服务商下拉选择
- 根据服务商动态显示 API Key placeholder
- 根据服务商动态显示可用模型列表

**单元测试**：
- 添加 3 个新测试用例
- 测试服务商管理、模型列表获取等
- 所有 26 个测试通过 ✅

#### 11. 浮动弹窗设计
**问题**：界面需要改为浮动弹窗形式，方便滑词询问，不聚焦时自动关闭

**分析**：
- 原设计是全屏窗口，不适合快速查询场景
- 需要小巧的浮动弹窗，类似词典工具
- 失焦自动关闭，避免占用屏幕空间

**技术实现**：
- Tauri 窗口配置：
  - `decorations: false` - 无边框
  - `transparent: true` - 透明背景
  - `alwaysOnTop: true` - 始置顶
  - `visible: false` - 默认隐藏
  - `skipTaskbar: true` - 不显示在任务栏
  - 尺寸：420x520（紧凑）

- 创建 `useWindow.ts` composable：
  - 窗口显示/隐藏控制
  - 失焦自动关闭（监听 `tauri://blur` 事件）
  - 窗口位置控制

- UI 设计：
  - 可拖动标题栏（`data-tauri-drag-region`）
  - 紧凑布局（小字体、小间距）
  - 可折叠的文本展示区
  - 设置面板覆盖式弹窗

**窗口行为**：
1. 默认隐藏
2. 复制文本或触发快捷键时显示
3. 失焦后自动隐藏
4. 可拖动标题栏移动位置
